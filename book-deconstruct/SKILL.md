# 书籍内容拆解

对任意书籍进行系统化拆解，输出完整、准确、易读的结构化分析报告。

## 触发条件

- 用户发送一本书（EPUB/PDF/TXT），或告知书籍路径
- 用户说"拆解这本书"、"分析这本书"、"这本书讲了什么"、"帮我快速了解这本书"
- 用户说"书内容报告"、"书籍拆解"

## 流程

### 1. 定位书籍

如果用户直接发文件，使用该文件。如果用户告知路径或书名：
- 先在 `$env:USERPROFILE\Desktop\書籍\` 搜索
- 再在 `$env:USERPROFILE\Downloads\` 搜索
- 支持模糊匹配书名（如用户说"如何阅读一本书"，匹配包含这些关键词的文件）
- 找不到时，主动询问路径

### 2. 提取文本

根据文件格式选择提取方式：

**EPUB：**
```powershell
pip install ebooklib beautifulsoup4 -q 2>$null
python scripts/extract_epub.py "<epub路径>" "<输出txt路径>"
```

**PDF：**
```powershell
pip install pymupdf -q 2>$null
python scripts/extract_pdf.py "<pdf路径>" "<输出txt路径>"
```

**TXT：** 直接读取。

### 3. 身份核验（提取后强制执行，防止张冠李戴）

**从文本内容中逐项提取，禁止用训练数据中的记忆替代：**

- [ ] **书名**（精确到副标题）：从文本前200行中提取原文
- [ ] **作者全名**：从文本标题或序言中提取
- [ ] **原书出版年份/版次**：从序言或版权页提取

提取后确认这三个信息与你的印象一致。如果不一致 → **以文本为准，不是以记忆为准。**

常见陷阱：同名书不同作者（如马歇尔《经济学原理》≠曼昆《经济学原理》），必须通过实际文本确认。

### 4. 分析内容（含防跳读强制记录）

文本提取后，按以下规则分批阅读。**这两条是强制的，不是建议：**

#### 4.1 分批规则

- 先扫描提取全书的目录/篇章结构（用脚本搜索章节标题正则），列出完整章节目录
- 每批 1500~2000 行，确保最终覆盖全书所有行
- **禁止**只读完前几章就跳到报告写作——目录中列出的每一章都必须出现在阅读记录中

#### 4.2 每批阅读后必须即时输出的记录（不可跳过）

每读完一批，立即输出以下格式的记录（作为防跳读的收据）：

```
━━━━━━━━━━━━━━━━━━━━
📖 阅读批次：第 N 批
📄 行号范围：Y ~ Z（共 X 行）
📑 覆盖章节：第 A 章 ~ 第 B 章，共 C 章
🔑 本批提取到的方法论/工具/核心观点：
  - [第A章] （观点/方法名）: （一句话描述）
  - [第B章] ...
━━━━━━━━━━━━━━━━━━━━
```

全书阅读结束后，汇总所有批次的章节数，与步骤4.1中提取的目录清单逐项对照。**任何一章没有出现在记录中 → 没有读完 → 必须补充阅读。**

### 5. 输出报告

生成完整的结构化 Markdown 报告。报告必须包含以下全部模块（按顺序，不可跳过任何模块）：

---

## 执行规则

1. **完整性优先**：提取全文后分批读取，确保覆盖全书每一章。分批记录是强制的，不是可选项。
2. **准确性**：所有观点引用必须基于书中实际内容，不要编造。作者名和书名从文本中提取，不凭记忆。
3. **结构清晰**：严格按模板输出，标题层级正确，不可跳过任何模块。
4. **易读性**：用中文表达，控制段落长度，善用列表和表格。
5. **批判性**：不要只是复述，要给出自己的分析和评估。
6. 报告默认直接保存到 Obsidian Vault：`E:\MyVault\书籍拆解\`，文件名为 `书名.md`。
7. 对于特别长的书（>500页），可以分两次输出：先输出前半部分（基本信息到核心观点），再输出后半部分（评估到行动清单）。

---

### 6. 🔍 强制自查（报告写入前必须执行，不可跳过）

**写完报告后，必须完成以下三步自查并输出自查证据表，确认无误后才能写入文件。**

**只需在报告末尾写上 `<!-- audit: passed -->` 是不算数的——三步自查的产出表格必须实际出现在报告中。**

---

#### 第一步：目录对照检查

将报告中的"章节拆解"部分与书中实际目录逐项对照，输出以下格式的表格：

```
### 🔍 自查一：目录对照

| 原书章节目录 | 报告中对应位置 | 状态 | 备注 |
|-------------|--------------|------|------|
| （逐章列出）  | 章节拆解-第X篇 | ✅ 或 ❌ | 如有遗漏：原因 |
```

**检查项：**
- [ ] 目录中的每一章是否都覆盖了？（有遗漏必须标注原因——是故意合并还是疏忽）
- [ ] 每一章的核心方法论/工具/观点是否被提取到对应位置？
- [ ] 序言中的关键信息（写作动机、全书结构说明）是否纳入？

→ **发现 ❌ 必须回补内容，全部 ✅ 才能进入第二步。**

---

#### 第二步：方法论深度检查

报告中的每个方法论/工具/框架，逐项检查四要素，输出以下格式的表格：

```
### 🔍 自查二：方法论深度

| 方法论 | 操作步骤 | 判断标准/阈值 | 案例/反例 | 适用范围 | 状态 |
|--------|---------|-------------|----------|---------|------|
| 方法一 | ✅ 步骤1-4 | ✅ XX参数 | ✅ XX案例 | ✅ XX条件 | ✅ |
| 方法二 | ✅ ... | ❌ 缺失 | ✅ ... | ✅ ... | ❌ |
```

**四要素标准：**

| 要素 | 检查问题 | 如果缺失 |
|------|---------|---------|
| **操作步骤** | 读者能照着步骤一步步做吗？ | 回去找书中步骤描述，补充进去 |
| **判断标准/阈值** | 有具体的数字、指标或条件吗？ | 如果书中有（如"<15%"），必须引用 |
| **案例或反例** | 有至少一个书中的真实案例吗？ | 找书中案例加入（反例优先——反例比正面例子更让人印象深刻） |
| **适用范围/局限** | 说明了什么情况下有效/无效吗？ | 如果原书有讨论适用条件，必须加入 |

→ **发现 ❌ 必须补充至通过。**

---

#### 第三步：全局一致性检查

逐项检查并输出结果：

```
### 🔍 自查三：全局一致性

- [ ] 框架自洽：[一句话概括的"A"] → [整体概要中展开为"B"] → [章节拆解逐篇支撑] → [核心观点中体现为"C"] → [行动清单对应"D"] ？ ✅
- [ ] 重点得当：篇幅最长的模块是《XX篇》，对应原书最核心的部分 [说明] ✅
- [ ] 读者测试：(a) 看完能说清核心观点？[简述] ✅ (b) 能用至少一个方法？[简述] ✅
```

---

### ⚠️ 三步自查完成后的最终步骤

1. 确认三步自查表中所有项均为 ✅
2. 在报告末尾加上 `<!-- audit: passed -->`
3. 写入 Obsidian Vault

**禁止**：未输出三步表格就写 `<!-- audit: passed -->`、跳过某一步、自查表中有 ❌ 仍然写入。

---

### ⚠️ 深度要求

**核心原则：报告不是目录摘要，而是让读者看完就能操作的知识库。**

对每个方法论/工具/框架，必须回答三个层次：
1. **是什么**（概念定义，一句话）
2. **怎么用**（具体操作步骤，含判断标准/阈值/公式）
3. **为什么**（底层假设或原理）

**对实用类书籍**：方法论部分是报告的核心，不是附加项。每个方法论必须有：操作步骤、判断标准/阈值、案例或反例、适用范围。优先使用书中的真实案例。

**对理论/思想类书籍**：核心观点要有逻辑链条（前提→论证→结论）。区分"作者的原创观点"和"作者引用的他人观点"。
